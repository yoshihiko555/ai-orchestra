# Design Skill — 初回応答

## Phase 判定

ユーザー入力: 「ECサイトにレビュー機能を追加したいので、要件定義から始めたい。既存のECサイトへの機能追加です。」

- 「要件定義から始めたい」→ **Phase 1（要件定義）から開始**
- 「既存のECサイトへの機能追加」→ **Phase 0（既存システム調査）を先行実施**

---

## Phase 0: 既存システム調査（自動実行中）

「既存のECサイトへの機能追加」とのことなので、まずプロジェクトのコードから技術スタックを自動調査します。

プロジェクト配下の設定ファイル（`package.json`, `pyproject.toml`, `go.mod`, `Gemfile`, `requirements.txt`, `docker-compose.yml` 等）を確認しています...

> 調査結果をもとに技術スタック・アーキテクチャ・既存の設計パターンを把握した上で、Phase 1 の要件定義に進みます。
> （既存プロジェクトのパスが別の場所にある場合は教えてください。自動で読み込みます。）

---

## Phase 1: 要件定義 — 開始

既存システムの技術スタック調査と並行して、要件定義を始めます。

まず、**プロジェクト概要と用語の整理**から入ります。レビュー機能の設計では「レビュー」「評価」「レーティング」といった用語が混在しやすいため、初期に用語集（glossary）を整備します。

---

## AskUserQuestion（初回質問）

以下について教えてください（2項目）:

**1. レビュー投稿者の範囲**
レビューを投稿できるのは誰ですか？

- a) ログインユーザーなら誰でも投稿可能
- b) その商品を購入したユーザーのみ投稿可能（購入済み限定）
- c) ゲストユーザー（ログイン不要）も投稿可能

**2. レビューの管理・モデレーション**
投稿されたレビューはどのように管理しますか？

- a) 投稿即時公開（モデレーションなし）
- b) 管理者が承認してから公開
- c) AI や自動フィルタで問題のある投稿を検出・非公開化

---

### 補足: 用語の事前整理（用語集の骨格）

設計を進める中で以下の用語を使います。認識がずれていれば修正してください。

| 用語 | 定義（仮） |
|------|----------|
| レビュー | 商品に対してユーザーが投稿する評価コンテンツ（星評価 + テキスト） |
| 星評価（レーティング） | 1〜5の整数値で商品を評価する数値指標 |
| レビュアー | レビューを投稿するユーザー |
| 総合評価 | 商品に紐づく全レビューの星評価の平均値 |
| レビュー公開 | レビューが他ユーザーに閲覧可能な状態になること |

---

### 次のステップ（進行予定）

回答をいただいた後、以下の順序で要件定義を進めます：

1. **プロジェクト概要の整理** — `docs/project-overview.md` 作成
2. **用語集の確定** — `docs/glossary.md` 作成
3. **機能一覧の作成** — `docs/requirements/feature-list.md`（MoSCoW分類付き）
4. **機能要件の詳細化** — `docs/requirements/functional.md`
5. **非機能要件の確認** — `docs/requirements/non-functional.md`

Phase 1 完了後、ご確認いただいた上で Phase 2（基本設計）に進みます。
