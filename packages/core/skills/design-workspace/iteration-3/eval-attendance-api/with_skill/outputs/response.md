# 勤怠管理システム API 設計

ご要件を確認しました。

- 出退勤打刻、勤怠一覧取得、月次レポート生成が主要機能
- 要件はほぼ固まっている → **Phase 2（基本設計）から開始**
- API設計とデータモデル設計に絞る → 画面設計はスキップ

進める前に、設計の前提を確認させてください。

---

## AskUserQuestion

以下の3点を教えてください。

**1. 新規開発か既存システムの改修か？**

- 新規でゼロから構築する
- 既存の勤怠システムを改修・リプレースする（既存コードベースあり）

→ 既存システムの場合は、現行のDB構造や技術スタックを調査してから設計に入ります（Phase 0）。

**2. 技術スタックの制約はありますか？**

例:
- 言語・フレームワーク（例: Python/FastAPI、Node.js/Express、Go/Gin）
- データベース（例: PostgreSQL、MySQL）
- 認証方式（例: JWT、セッション、社内IdPとのSAML/OIDC連携）

既に決まっているものがあれば教えてください。未決定の場合はこちらで提案します。

**3. 利用者の種類と権限の想定は？**

例:
- 一般社員（自分の勤怠を打刻・確認）
- 管理者（部下の勤怠確認・月次レポート生成）
- 経理/人事（全社の月次レポート閲覧）

誰がどの操作をできる必要があるかで、API の認可設計が変わります。
