# Design Skill — Initial Response

## タスク

フィットネス記録アプリをiOSとAndroid向けに新規開発したい。ワークアウトの記録、体重・体脂肪率のトラッキング、目標設定と進捗表示ができるアプリ。設計をお願いしたい。

- **開始フェーズ**: Phase 1（要件定義）から開始
- **プラットフォーム**: iOS / Android ネイティブアプリ（ネイティブアプリ向けの設計項目を適用）

## Phase 1: 要件定義 — 開始

フィットネス記録アプリの設計を進めます。
要件定義を始めるにあたり、プロジェクトの前提を確認します。

## AskUserQuestion（第1ターン）

> 以下の2点を教えてください。
>
> **1. ターゲットユーザーと主な利用シーン**
> アプリを使う人はどんな人を想定していますか？
> 例:
> - フィットネス初心者（週1〜2回ジムに通う人）
> - 中〜上級者（毎日トレーニングするアスリート志向）
> - ダイエット目的メイン（体重・体脂肪管理がメイン）
>
> **2. バックエンド・データ同期の方針**
> データはどこに保持しますか？
> - ローカル端末のみ（クラウド同期なし）
> - クラウド同期あり（iOS/Android間でデータ共有、複数端末対応）
> - ユーザー認証（アカウント登録・ログイン）が必要か

## 進行計画

回答内容を踏まえて以下の順序で要件定義を進めます。

### Phase 1 進行ステップ

1. **第1ターン（本ターン）**: ターゲットユーザー・データ同期方針の確認
2. **第2ターン**: 機能スコープの詳細確認（Must/Should/Could の仕分け）
3. **第3ターン**: 非機能要件の確認（パフォーマンス・オフライン対応・通知等）
4. **ドキュメント作成**: 合意した内容を以下のファイルに出力

### Phase 1 成果物（受け入れチェックリスト）

- [ ] `docs/project-overview.md` — プロジェクト概要（目的・背景・スコープ）
- [ ] `docs/glossary.md` — 用語集（ワークアウト・セット・レップ・体脂肪率等）
- [ ] `docs/requirements/feature-list.md` — 機能一覧（ID付き、Must/Should/Could分類）
- [ ] `docs/requirements/functional.md` — Must機能の要件詳細（ユースケース・受け入れ条件）
- [ ] `docs/requirements/non-functional.md` — 非機能要件（オフライン対応・パフォーマンス・セキュリティ等）
- [ ] 全機能に優先順位（Must/Should/Could）が付けられている
- [ ] スコープ（In/Out）が明確化されている

### Phase 2 以降の予定（概要）

Phase 1完了後、ネイティブアプリ向けの設計項目を実施します。

| フェーズ | 主な成果物 |
|---------|-----------|
| Phase 2: 基本設計 | アーキテクチャ（クロスプラットフォーム vs ネイティブ選定）、画面一覧・遷移、API一覧、データモデル |
| Phase 3: 詳細設計 | 各画面の詳細設計書、API個別設計書、テーブル定義 |

> **プラットフォーム考慮点**: iOS/Androidネイティブアプリのため、Phase 2では以下も設計対象になります。
> - クロスプラットフォーム戦略（React Native / Flutter / ネイティブ別開発 等）
> - プラットフォーム固有のUI/UXガイドライン（iOS: Human Interface Guidelines / Android: Material Design）
> - ヘルスケアデータ連携（Apple Health / Google Fit）の要否
