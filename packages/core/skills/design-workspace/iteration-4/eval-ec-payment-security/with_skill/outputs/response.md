# Design スキル — 第1ターン応答

## タスクプロンプト

> オンライン決済機能付きのECサイトを新規構築したい。クレジットカード情報を扱うので、セキュリティ面もしっかり設計したい。要件定義から始めたい。

---

## Phase 判定

- 「要件定義から始めたい」→ **Phase 1: 要件定義** から開始
- 新規プロジェクト → Phase 0（既存システム調査）はスキップ
- 「クレジットカード情報を扱う」「セキュリティ面もしっかり設計したい」→ **セキュリティ設計トラック（拡張）** のトリガーを検出

## セキュリティ設計トラックの提案

`references/security-design.md` を読み込み済み。

「クレジットカード情報を扱う」という明示があるため、このプロジェクトは以下に該当します：

- 決済情報（機密データ）を扱う
- PCI DSS 準拠が必要になる可能性がある
- セキュリティ設計トラックのトリガー条件を満たしている

→ Phase 1 の非機能要件ヒアリングと並行して、セキュリティ設計（`docs/security/`）を実施することを提案します。


## スキル実行の内部メモ

### 読み込んだ参照ファイル

- `references/requirements.md` — Phase 1 の詳細手順
- `references/security-design.md` — セキュリティ設計トラックの手順

### トリガー検出

| 条件 | 検出値 |
|------|--------|
| 機密データ | クレジットカード情報（明示） |
| セキュリティへの言及 | 「セキュリティ面もしっかり設計したい」（明示） |
| 拡張トラック提案 | セキュリティ設計トラック → **提案を実施** |

### 想定する設計フロー（初回判断）

```
Phase 1: 要件定義
  + セキュリティ設計トラック（並行）
      └─ Step 1: セキュリティ要件整理（PCI DSS への言及）
      └─ Step 2: 脅威モデリング（STRIDE）
      └─ Step 3: 認証・認可アーキテクチャ
      └─ Step 4: データ保護設計
    ↓
Phase 2: 基本設計
Phase 3: 詳細設計
```

### 次ターンで確認すること（予定）

- 決済代行サービスの利用有無（→ PCI DSS スコープの確定に直結）
- 商品カテゴリ・事業規模（→ スコープ設定）
- ターゲットユーザー（→ 認証要件、ユーザーロール定義）
