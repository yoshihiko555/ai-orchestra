# オンライン決済機能付きECサイト — 要件定義

## 1. プロジェクト概要

クレジットカード決済を含むECサイトを新規構築する。
PCI DSS 準拠を前提とし、セキュリティを設計の中心に置く。

---

## 2. ビジネス要件

### 2.1 主要機能

| 機能カテゴリ | 要件 |
|------------|------|
| 商品管理 | 商品一覧・詳細表示、在庫管理、カテゴリ分類 |
| ユーザー管理 | 会員登録・ログイン・プロフィール管理 |
| カート・注文 | カート操作、注文確定、注文履歴 |
| 決済 | クレジットカード決済、決済履歴、返金対応 |
| 管理画面 | 商品・注文・ユーザー管理、売上レポート |

### 2.2 想定ユーザー

- エンドユーザー（購入者）
- 店舗管理者（商品登録・注文管理）
- システム管理者（インフラ・セキュリティ管理）

---

## 3. 機能要件

### 3.1 認証・アカウント管理

- メールアドレス + パスワードによる会員登録
- OAuth 2.0 によるソーシャルログイン（Google / Facebook）
- メールアドレス確認（本人確認メール送信）
- パスワードリセット機能
- 多要素認証（MFA）のオプション提供
- セッション管理（同一アカウントの複数端末ログイン制御）

### 3.2 商品・カート

- 商品一覧（ページネーション・フィルタ・ソート）
- 商品詳細（画像複数枚、バリエーション：サイズ・色）
- カートへの追加・削除・数量変更
- お気に入り（ウィッシュリスト）
- 在庫数リアルタイム表示

### 3.3 決済フロー

- クレジットカード決済（Visa / Mastercard / JCB / AMEX）
- 決済代行サービス経由（Stripe または PAY.JP）
- カード情報はサーバーに保存しない（トークン化）
- 3-D セキュア 2.0 対応
- 注文確定後の確認メール自動送信
- 返金・キャンセル処理

### 3.4 注文・配送

- 注文ステータス管理（注文受付 → 決済完了 → 発送準備 → 発送済 → 配達完了）
- 配送先住所の複数登録
- 配送業者 API 連携（追跡番号連携）

---

## 4. 非機能要件

### 4.1 パフォーマンス

- ページロード: 3 秒以内（TTI）
- 決済処理: 10 秒以内
- 同時接続: 1,000 ユーザー以上を想定

### 4.2 可用性

- SLA: 99.9%（月間ダウンタイム 43 分以内）
- 定期メンテナンスは深夜帯（0:00〜4:00）に実施

### 4.3 スケーラビリティ

- セール時（通常の 10 倍トラフィック）への対応
- 水平スケールアウト可能なアーキテクチャ

---

## 5. セキュリティ要件（最重要）

### 5.1 PCI DSS 準拠

クレジットカード情報を扱うため、PCI DSS（Payment Card Industry Data Security Standard）への準拠が必須。

**基本方針: カード情報をサーバーに持たない（SAQ A 相当）**

- カード番号（PAN）、CVV、有効期限をサーバーに保存しない
- 決済代行サービス（Stripe / PAY.JP）のクライアントサイド SDK でトークン化
- トークンのみをサーバーで管理

### 5.2 通信セキュリティ

- 全通信を HTTPS（TLS 1.2 以上）に強制
- HSTS（HTTP Strict Transport Security）を有効化
- 証明書は自動更新（Let's Encrypt または ACM）
- Content Security Policy (CSP) の設定

### 5.3 認証・セッション

- パスワードは bcrypt（コストファクター 12 以上）でハッシュ化
- セッション ID はランダム生成（128 bit 以上のエントロピー）
- セッション有効期限の設定（アイドル 30 分 / 最大 24 時間）
- ログアウト時のセッション完全無効化
- ブルートフォース対策（ログイン試行回数制限、アカウントロック）

### 5.4 入力検証・インジェクション対策

- すべての入力値のサーバーサイドバリデーション
- SQL インジェクション対策（ORM 使用 + パラメータバインディング）
- XSS 対策（出力エスケープ、CSP）
- CSRF 対策（CSRF トークン）
- XXE / SSRF 対策

### 5.5 アクセス制御

- RBAC（Role-Based Access Control）の導入
  - 購入者 / 店舗管理者 / システム管理者の 3 ロール
- 管理画面への IP 制限
- API エンドポイントの認可チェック（全エンドポイント）

### 5.6 ログ・監査

- アクセスログ・エラーログ・決済ログの分離保存
- ログに個人情報・カード情報を含めない（マスキング）
- 不審なアクセスのアラート通知（SIEM 連携 or シンプルなルールベース）
- ログ保存期間: 1 年以上（PCI DSS 要件）

### 5.7 インフラセキュリティ

- WAF（Web Application Firewall）の導入
- DDoS 対策（CDN / クラウドプロバイダー機能）
- 脆弱性スキャンの定期実施
- 依存パッケージの定期更新（Dependabot 等）
- 本番環境への直接アクセス禁止（踏み台サーバー経由）

## 6. システムアーキテクチャ方針

### 6.1 推奨スタック

| レイヤー | 技術選定候補 |
|---------|------------|
| フロントエンド | Next.js (React) |
| バックエンド | Node.js (NestJS) または Python (FastAPI) |
| データベース | PostgreSQL |
| キャッシュ | Redis |
| ストレージ | AWS S3 / GCS |
| 決済 | Stripe または PAY.JP |
| インフラ | AWS / GCP（コンテナ: ECS or Cloud Run） |
| CDN | CloudFront / Cloudflare |

### 6.2 決済フロー（トークン化）

```
ユーザー
  ↓ カード情報入力（決済代行のiframe/SDK内）
決済代行サービス（Stripe等）
  ↓ トークン発行
フロントエンド
  ↓ トークンをサーバーに送信（カード情報は含まない）
バックエンド
  ↓ トークンで決済代行APIを呼び出し
決済代行サービス
  ↓ 決済結果を返却
バックエンド → ユーザーに結果通知
```

**ポイント: カード情報がサーバーを経由しない構成にすることで、PCI DSS スコープを最小化する。**

---

## 7. 次のステップ

要件定義確認後、以下の順序で設計を進めることを推奨します。

1. **アーキテクチャ設計** — システム全体構成図、コンポーネント定義
2. **データモデル設計** — ER 図（ユーザー・商品・注文・決済）
3. **API 設計** — エンドポイント一覧、リクエスト/レスポンス仕様
4. **認証・認可設計** — JWT / セッション方式の選定、RBAC 詳細
5. **セキュリティ設計書** — 脅威モデリング（STRIDE）、対策一覧
6. **インフラ設計** — 本番・ステージング環境構成

---

## 8. 確認事項（ヒアリング推奨）

設計を具体化するにあたり、以下を確認することを推奨します。

- 想定月間注文件数・ユーザー数の規模感
- 決済代行サービスの選定方針（Stripe / PAY.JP / その他）
- 対応する決済手段の追加予定（コンビニ払い、銀行振込、Apple Pay 等）
- 開発体制とスケジュール（MVP リリース目標時期）
- 既存システムとの連携有無（在庫管理システム、会計システム等）
- 個人情報保護方針（データ保存場所、データ主権の要件）
