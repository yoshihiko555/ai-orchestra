# ECサイト レビュー機能 — 要件定義・設計

## 1. 要件定義

### 1.1 背景・目的

既存 EC サイトに商品レビュー機能を追加し、購買判断の補助および信頼性向上を図る。

---

### 1.2 ユーザーストーリー

| # | ロール | したいこと | 目的 |
|---|--------|-----------|------|
| 1 | 購入済みユーザー | 商品に星評価（1〜5）とテキストでレビューを投稿したい | 他のユーザーの参考にしたい |
| 2 | 閲覧ユーザー | 商品ページでレビュー一覧と平均評価を見たい | 購買判断に活用したい |
| 3 | 購入済みユーザー | 自分が投稿したレビューを編集・削除したい | 内容を修正したい |
| 4 | 管理者 | 不適切なレビューを非表示・削除したい | サイトの品質を保ちたい |

---

### 1.3 機能要件

#### レビュー投稿

- **投稿できるユーザー**: ログイン済みかつ対象商品を購入済みのユーザー（注文ステータス: 配送完了以降）
- **評価**: 星 1〜5 の整数値（必須）
- **テキスト**: 任意入力（最大 1,000 文字）
- **タイトル**: 任意入力（最大 100 文字）
- **1 商品につき 1 ユーザー 1 件まで**（重複投稿不可）
- 投稿後に編集・削除が可能（投稿者本人のみ）

#### レビュー表示

- 商品詳細ページに平均星評価と件数を表示
- 星ごとの内訳（ヒストグラム）を表示
- レビュー一覧をページネーションで表示
  - デフォルトソート: 投稿日時の新しい順
  - ソート切り替え: 評価の高い順 / 低い順
- 各レビューに投稿者ニックネーム・投稿日時・評価・テキストを表示

#### 管理機能

- 管理画面でレビュー一覧を検索・フィルタリング
- レビューの非表示・再表示・削除操作
- 不適切レポート機能（閲覧ユーザーが通報可能）

---

### 1.4 非機能要件

| 項目 | 要件 |
|------|------|
| パフォーマンス | 商品ページのレビュー初回表示 < 500ms（キャッシュ適用後） |
| スケーラビリティ | 1 商品あたり数万件のレビューに対応 |
| セキュリティ | XSS 対策（テキストエスケープ）、CSRF 対策、認証必須 |
| 可用性 | 既存機能に影響を与えない段階的デプロイ |
| スパム対策 | レートリミット（1 ユーザー・1 時間に 5 件まで投稿） |

---

### 1.5 スコープ外（今回の対象外）

- 画像・動画のレビュー添付
- レビューへの「役に立った」投票
- レビューへのコメント返信
- SNS シェア機能

---

## 2. 設計

### 2.1 データモデル

#### `reviews` テーブル

| カラム | 型 | 説明 |
|--------|----|------|
| `id` | UUID / BIGINT | 主キー |
| `product_id` | FK → products | 対象商品 |
| `user_id` | FK → users | 投稿者 |
| `order_id` | FK → orders | 購入証明（購入済み検証用） |
| `rating` | TINYINT (1-5) | 星評価 |
| `title` | VARCHAR(100) | レビュータイトル（nullable） |
| `body` | TEXT | レビュー本文（nullable） |
| `status` | ENUM('published', 'hidden') | 表示状態 |
| `created_at` | TIMESTAMP | 投稿日時 |
| `updated_at` | TIMESTAMP | 更新日時 |

**インデックス:**
- `(product_id, status, created_at)` — 商品別レビュー一覧取得
- `(user_id, product_id)` — 重複チェック（UNIQUE 制約）
- `(product_id, rating)` — 平均評価・件数集計

#### `review_reports` テーブル（通報機能）

| カラム | 型 | 説明 |
|--------|----|------|
| `id` | UUID / BIGINT | 主キー |
| `review_id` | FK → reviews | 対象レビュー |
| `reporter_user_id` | FK → users | 通報者 |
| `reason` | VARCHAR(255) | 通報理由 |
| `created_at` | TIMESTAMP | 通報日時 |

### 2.2 API 設計

#### エンドポイント一覧

| メソッド | パス | 概要 | 認証 |
|---------|------|------|------|
| `GET` | `/api/v1/products/{id}/reviews` | レビュー一覧取得 | 不要 |
| `GET` | `/api/v1/products/{id}/reviews/summary` | 平均評価・件数取得 | 不要 |
| `POST` | `/api/v1/products/{id}/reviews` | レビュー投稿 | 必要（購入済み） |
| `PUT` | `/api/v1/reviews/{review_id}` | レビュー編集 | 必要（投稿者） |
| `DELETE` | `/api/v1/reviews/{review_id}` | レビュー削除 | 必要（投稿者） |
| `POST` | `/api/v1/reviews/{review_id}/reports` | 通報 | 必要（ログイン済み） |
| `GET` | `/api/v1/admin/reviews` | 管理用一覧 | 必要（管理者） |
| `PATCH` | `/api/v1/admin/reviews/{review_id}/status` | ステータス変更 | 必要（管理者） |

#### `GET /api/v1/products/{id}/reviews` レスポンス例

```json
{
  "data": [
    {
      "id": "uuid-xxx",
      "rating": 4,
      "title": "使いやすい",
      "body": "品質が良く、満足しています。",
      "author": {
        "nickname": "taro_yamada"
      },
      "created_at": "2026-03-01T10:00:00Z"
    }
  ],
  "pagination": {
    "page": 1,
    "per_page": 20,
    "total": 142
  }
}
```

#### `POST /api/v1/products/{id}/reviews` リクエスト

```json
{
  "rating": 5,
  "title": "最高の商品",
  "body": "期待以上でした。リピート確定です。"
}
```

### 2.3 フロントエンド設計

#### コンポーネント構成（商品詳細ページへの追加）

```
ProductDetailPage
├── ReviewSummary          // 平均評価 + 星ヒストグラム
├── ReviewForm             // 投稿フォーム（購入済みユーザーのみ表示）
│   ├── StarRatingInput    // 星選択 UI
│   ├── TextInput (title)
│   └── TextArea (body)
└── ReviewList             // レビュー一覧
    ├── SortSelector       // 新しい順 / 評価高い順 / 低い順
    ├── ReviewCard (×n)    // 個別レビュー表示
    └── Pagination
```

#### 状態管理のポイント

- レビュー一覧・サマリーはサーバーサイドキャッシュ（Redis / CDN）で高速化
- 投稿・編集後はクライアント側で楽観的更新（Optimistic UI）
- 投稿資格チェック（購入済み判定）は初回ページロード時に API で確認

---

### 2.4 購入済み検証フロー

```
ユーザーがレビュー投稿ボタン押下
  ↓
POST /api/v1/products/{id}/reviews
  ↓
サーバー: orders テーブルを検索
  └─ user_id + product_id + status = '配送完了' のレコードが存在するか確認
  ↓
存在する → reviews テーブルに INSERT
存在しない → 403 Forbidden ("購入済みユーザーのみレビューを投稿できます")
```

---

### 2.5 パフォーマンス設計

- **平均評価・件数**: `product_id` ごとに Redis にキャッシュ（TTL: 5 分）
  - レビュー投稿・削除・ステータス変更時にキャッシュを破棄
- **レビュー一覧**: カーソルベースページネーションで大量件数に対応（`created_at` + `id` でカーソル設計）
- **N+1 防止**: レビュー一覧取得時にユーザー情報を JOIN で一括取得

|---------|------|--------|

## 4. 未確定事項・確認が必要な点

以下は設計を進める前に確認が必要な項目です。

1. **技術スタック**: 既存 EC サイトのバックエンド言語・フレームワーク（Rails / Django / Laravel / Node.js 等）
2. **認証方式**: セッション認証 / JWT トークン認証のどちらか
3. **DB**: MySQL / PostgreSQL 等（UUID vs BIGINT の選択に影響）
4. **購入済み判定の基準**: 「配送完了」以外に「注文確定時点」から投稿可能にするか
5. **匿名レビュー**: ニックネーム非表示オプションを許可するか
6. **レビュー編集の制限**: 投稿後 XX 日以内のみ編集可能などの制限を設けるか
7. **キャッシュインフラ**: Redis が利用可能か
