# 大規模社内ポータルサイト 基本設計・UI統一ルール

## 1. プロジェクト概要整理

### 前提条件
- 規模: 20画面以上のWebアプリケーション
- 体制: フロントエンドエンジニア5名による並行開発
- フェーズ: 基本設計（画面設計 + UIルール策定）

### 並行開発の主なリスク
- UIの実装が担当者ごとにバラバラになる（見た目・挙動の不一致）
- コンポーネントの重複実装による工数ロス
- 命名規則・ファイル構成の不統一によるマージコスト増大
- 共通コンポーネントの変更が予期せぬ箇所に影響する

---

## 2. 推奨アプローチ: デザインシステム先行策定

5名が並行開発する場合、**デザインシステム（コンポーネントライブラリ + UIルール）を最初に確立する**ことが最重要。

### フェーズ分割案

```
Phase 1（1〜2週間）: 基盤整備
  ├─ デザイントークン定義
  ├─ 共通コンポーネント設計
  ├─ ファイル構成・命名規則策定
  └─ 開発環境セットアップ（Storybook等）

Phase 2（2〜3週間）: 画面設計
  ├─ 画面一覧・遷移図作成
  ├─ ワイヤーフレーム作成（全20画面）
  └─ 担当画面の割り当て

Phase 3以降: 並行実装
  └─ Phase 1の基盤を使って各自が担当画面を実装
```

### 3-2. コンポーネント設計方針

#### Atomic Design の適用（簡略版）

| 階層 | 内容 | 例 |
|------|------|----|
| Atoms | それ以上分割できない最小単位 | Button, Input, Badge, Icon |
| Molecules | Atoms の組み合わせ | SearchBar, FormField, Card |
| Organisms | 機能を持つUIブロック | Header, Sidebar, DataTable |
| Templates | ページのレイアウト構造 | DashboardLayout, FormLayout |
| Pages | 実際の画面 | UserListPage, SettingsPage |

#### コンポーネント設計の原則

- **Props で挙動を制御**: variant, size, disabled 等で状態を表現
- **スタイルの上書き禁止**: 外部から直接スタイルを上書きする実装を避ける
- **アクセシビリティ必須**: aria-label, role, keyboard navigation を標準実装
- **ローディング状態を持つ**: データ取得を伴うコンポーネントは skeleton/spinner を内包

### 3-3. ファイル構成規則

```
src/
├─ components/
│   ├─ atoms/
│   │   ├─ Button/
│   │   │   ├─ Button.tsx         # コンポーネント本体
│   │   │   ├─ Button.stories.tsx # Storybook
│   │   │   ├─ Button.test.tsx    # ユニットテスト
│   │   │   └─ index.ts           # export
│   │   └─ Input/
│   ├─ molecules/
│   ├─ organisms/
│   └─ templates/
├─ pages/                 # 画面単位のコンポーネント
├─ hooks/                 # カスタムフック
├─ stores/                # 状態管理
├─ utils/                 # ユーティリティ関数
├─ types/                 # 型定義
├─ constants/             # 定数
└─ styles/
    ├─ tokens.css         # デザイントークン
    └─ global.css         # グローバルスタイル
```

### 3-4. 命名規則

| 対象 | 規則 | 例 |
|------|------|----|
| コンポーネントファイル | PascalCase | `UserCard.tsx` |
| コンポーネント名 | PascalCase | `export const UserCard` |
| カスタムフック | camelCase + use接頭辞 | `useUserList.ts` |
| ユーティリティ関数 | camelCase | `formatDate.ts` |
| 型定義 | PascalCase + 用途サフィックス | `UserListProps`, `ApiResponse` |
| 定数 | UPPER_SNAKE_CASE | `MAX_PAGE_SIZE` |
| CSSクラス | kebab-case | `.user-card__title` |

### 3-5. 共通UIパターン

#### フォームのバリデーション表示
- エラーメッセージはフィールド直下に赤字で表示
- 必須フィールドには「*（必須）」ラベルを付与
- Submit 後のみエラー表示（入力中はリアルタイム非表示）

#### テーブル（一覧画面）
- ソート可能列にはアイコンを表示
- ページネーションは「前/次 + ページ番号」形式
- 空データ時は「データがありません」のメッセージと追加導線を表示
- 行クリックで詳細画面に遷移（選択行のハイライト付き）

#### モーダル
- 背景オーバーレイは半透明黒（rgba(0,0,0,0.5)）
- Esc キーで閉じる
- フォーカストラップを実装（Tab キーがモーダル外に出ない）
- 破壊的操作（削除等）は確認モーダルを必ず挟む

#### 通知（Toast）
- 右上固定で表示、3秒後に自動消去
- 成功（緑）/ 失敗（赤）/ 警告（黄）/ 情報（青）の4種類
- 手動で閉じるボタンも提供

#### ローディング
- ページ全体のローディングはフルスクリーンスピナー
- テーブル等のパーツはスケルトンスクリーンを使用
- ボタン押下後の処理中はボタンを disabled + スピナーに変更

## 4. 画面設計の進め方

### 4-1. 画面一覧の整理（先に決める）

まず全画面をリストアップし、以下の情報を整理する。

| # | 画面名 | URL | 権限 | データソース | 担当者 |
|---|--------|-----|------|-------------|--------|
| 1 | ダッシュボード | /dashboard | 全員 | API: /summary | - |
| 2 | ユーザー一覧 | /users | 管理者 | API: /users | - |
| ... | ... | ... | ... | ... | - |

### 4-2. 画面分類と共通レイアウト

20画面以上をレイアウトパターンで分類する。

| パターン | 適用画面例 |
|----------|-----------|
| ダッシュボードレイアウト（サイドナビ + メインコンテンツ） | ダッシュボード、各一覧画面 |
| フォームレイアウト（中央寄せ + 左右余白） | 新規作成、編集画面 |
| 詳細レイアウト（2カラム: 基本情報 + サイドパネル） | 詳細表示画面 |
| 認証レイアウト（ナビなし + センター配置） | ログイン、パスワードリセット |

### 4-3. 画面ワイヤーフレームの優先順位

以下の順序でワイヤーフレームを作成し、レビューを進める。

1. **共通レイアウト**（ヘッダー・サイドバー・フッター）
2. **最頻出画面**（一覧画面・詳細画面の代表例）
3. **複雑なフォーム画面**（入力項目が多い画面）
4. **その他の画面**

## 5. 5名チームの役割分担案

### 推奨体制

| 役割 | 人数 | 担当内容 |
|------|------|---------|
| コンポーネントリード | 1名 | デザインシステム管理、共通コンポーネント実装、コードレビュー |
| 画面実装エンジニア | 4名 | 担当画面の実装（各自4〜5画面） |

### コンポーネントリードの責務
- `components/atoms/`, `components/molecules/` の設計・実装
- Storybook の管理・更新
- PRレビューでUIルール遵守を確認
- 「どのコンポーネントを使うか」の相談窓口

### 並行開発のルール
- 共通コンポーネントを変更する場合は必ずリードに確認
- 既存コンポーネントで要件を満たせない場合は新規追加をリードと合意
- スタイルの直書き（インラインスタイル）は原則禁止

## 6. 技術スタック推奨（参考）

以下は一般的な選択肢。既存の社内標準があればそちらを優先する。

| カテゴリ | 推奨候補 | 理由 |
|----------|---------|------|
| フレームワーク | Next.js (React) | 豊富なエコシステム、SSR/SSG対応 |
| UIコンポーネント | 自社製 or shadcn/ui | カスタマイズ自由度が高い |
| 状態管理 | Zustand / TanStack Query | シンプルで学習コストが低い |
| スタイリング | Tailwind CSS or CSS Modules | チームの習熟度に合わせて選択 |
| コンポーネントカタログ | Storybook | UI確認・ドキュメント化に必須 |
| フォーム | React Hook Form + Zod | バリデーション込みで扱いやすい |

---

## 7. すぐに着手すべきアクション

1. **画面一覧の確定**: 全画面を洗い出し、URLと権限を定義する
2. **デザイントークンの合意**: カラーパレット・余白・フォントをデザイナーと確定する
3. **共通コンポーネントの優先度付け**: 最初に作る10〜15コンポーネントをリストアップ
4. **Storybook環境の構築**: 週1でコンポーネント状況を全員が確認できる場を作る
5. **コーディング規約のドキュメント化**: ESLint/Prettier設定と合わせてREADMEに記載
6. **レビュープロセスの決定**: PR単位のレビュワー、マージ条件を明文化する
