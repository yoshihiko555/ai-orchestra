# ADR-013: PR 作成スキル追加と issue-workflow パッケージの git-workflow への改名

- **ステータス**: accepted
- **日付**: 2026-04-09
- **決定者**: yoshihiko
- **関連 Issue**: #35

## コンテキスト

`issue-workflow` パッケージは `issue-create` / `issue-fix` の 2 スキルを提供しており、PR 作成は `issue-fix` の Phase 4-6 に内包されている。以下の課題がある:

1. Issue を経由せずに PR だけ作りたいユースケースに対応できない
2. PR 作成ロジックが `issue-fix` に密結合しており再利用できない
3. パッケージ名 `issue-workflow` が実際の責務（Issue + PR + Git ワークフロー全体）と乖離している

## 検討した選択肢

### 選択肢 A: pr-create スキルのみ追加（パッケージ名は維持）

- メリット: 影響範囲が小さい、後方互換性を維持
- デメリット: パッケージ名と実態の乖離が拡大する

### 選択肢 B: pr-create スキル追加 + パッケージ名を git-workflow に改名

- メリット: 責務と名称が一致する、今後の拡張（commit スキル等）にも対応しやすい
- デメリット: 影響範囲が広い（15+ 箇所のリネーム）、既存プロジェクトでの互換性破壊

### 選択肢 C: pr-create を別パッケージとして独立

- メリット: 既存パッケージに手を入れない
- デメリット: 関連スキルが分散する、パッケージ間の依存が増える

## 決定

**選択肢 B** を採用する。

理由:
- `pr-create` 追加により責務が Issue 以外に広がるため、名称を合わせるのが自然
- 既存パッケージの命名規則（機能領域名）に沿っている
- ai-orchestra 自体がまだ初期段階のため、互換性破壊の影響は限定的
- `issue-fix` の PR 作成部分を `pr-create` スキルとして切り出し、composition で再利用可能にする

## 実装方針

1. `facets/instructions/pr-create.md` を新規作成（`.github` リポの PR テンプレート・git-release-policy を反映）
2. `facets/compositions/pr-create.yaml` を新規作成
3. `issue-fix` の Phase 4-6 の PR 作成ロジックを PR Standards Policy 参照に簡素化（スキル間呼び出しは facet 設計思想に合わないため、policy 共有方式を採用）
4. `packages/issue-workflow/` を `packages/git-workflow/` にリネーム
5. manifest.json の `skills` に `pr-create` を追加
6. 全参照箇所（docs, config, orchestra.json 等）を更新

## 影響

- `packages/issue-workflow/` → `packages/git-workflow/` ディレクトリ移動
- `.claude/config/issue-workflow/` → `.claude/config/git-workflow/` ディレクトリ移動
- `.claude/orchestra.json` のパッケージ名・config パス更新
- `README.md`, `packages/README.md`, `docs/` 内の参照更新
- 既存プロジェクトで `issue-workflow` を install 済みの場合、`orchex` の再 init が必要
