# ADR-20260315-010: Faceted Prompting によるスキル・ルールのファセット分解と自動生成

- **ステータス**: accepted（実装・検証完了）
- **日付**: 2026-03-15
- **決定者**: yoshihiko
- **関連**: faceted-prompting リポジトリ (https://github.com/nrslib/faceted-prompting)

## コンテキスト

14スキル・12ルールを `.claude/skills/` と `.claude/rules/` で管理しているが、以下の課題がある:

1. **ルールの重複**: coding-principles ルールと simplify スキルに同じコード品質ルールが直書きされている
2. **Policy の散在**: 「推測と事実を区別」「AskUserQuestion 必須」等の共通ルールが複数スキルに個別記載
3. **クロス CLI 配布不可**: スキルは Claude Code 専用形式で、Codex CLI（`.codex/skills/`）への配布手段がない

nrs 氏の Faceted Prompting（プロンプトを Persona / Policy / Knowledge / Instruction / Output Contract に分解する手法）を参考に、スキル管理の見直しを検討した。

## 検討した選択肢

### 選択肢 A: faceted-prompting npm ツールをそのまま採用

- メリット: 実装済みの compose / install コマンドが使える
- デメリット: Knowledge がプロジェクト固有で管理困難、npm 依存の追加、既存 sync-orchestra.py との統合コスト

### 選択肢 B: faceted-prompting の設計思想を取り入れ、Python で同等機能を構築

- メリット: 既存の orchestra-manager.py / sync-orchestra.py と自然に統合できる
- メリット: composition YAML → SKILL.md 生成で、同じファセットからスキルとルールの両方を生成可能
- メリット: `--target codex` で Codex CLI 向けにも同一内容を配布可能
- デメリット: 自前実装のメンテナンスコスト

### 選択肢 C: SKILL.md 内にファイル参照を記述し、実行時に Claude が読み込む方式

- メリット: ビルドステップ不要
- デメリット: ファセットの読み込みが Claude の判断に依存し不確実、Codex/Gemini への配布に対応できない

## 決定

**選択肢 B** を採用。

### 実装した機能

1. **ファセットの分類**: Policy（4種）と Output Contract（3種）を共通部品として管理
   - Policies: code-quality, cli-language, dialog-rules, factual-writing
   - Output Contracts: tiered-review, compare-report, deep-dive-report
   - Persona / Knowledge / Instruction はスキル固有のため分離対象外

2. **ディレクトリ構成**: ai-orchestra リポジトリのトップレベルに `facets/` を配置
   ```
   facets/
   ├── policies/          # 4つの共有 Policy
   ├── output-contracts/  # 3つの共有 Output Contract
   ├── instructions/      # 26個のスキル・ルール固有 instruction
   └── compositions/      # 26個の組み立て定義 YAML
   ```

3. **`facet build` コマンド**: composition YAML + ファセット .md → SKILL.md / ルール .md を自動生成
   - `--target claude`: `.claude/skills/` および `.claude/rules/` に出力
   - `--target codex`: `.codex/skills/` および `.codex/rules/` に出力
   - Claude Code と Codex は同一フォーマット（SKILL.md）

4. **`facet extract` コマンド**: 生成済み SKILL.md を直接編集した後、instruction 部分をソースに書き戻す

5. **composition YAML スキーマ**:
   - `name`: 構成名（必須）
   - `type`: `skill`（frontmatter 付き）/ `rule`（frontmatter なし）
   - `package`: 関連パッケージ名（未インストール時はビルドスキップ + 既存生成物を削除）
   - `policies[]`: 参照する Policy ファセット
   - `output_contracts[]`: 参照する Output Contract ファセット
   - `instruction`: インライン文字列またはファイル参照（100文字以下 → ファイル参照として解決）
   - `frontmatter`: 生成 SKILL.md に付与する YAML frontmatter（skill のみ）

6. **SessionStart 自動実行**: sync-orchestra.py が facets/ を同期後に facet build を自動実行
   - 変更検知あり（mtime 比較でソースが新しい場合のみ実行）
   - `codex-suggestions` パッケージがインストール済みの場合は codex ターゲットも自動ビルド

7. **プロジェクトローカル対応**: 配布先プロジェクトの `.claude/facets/` に独自の instruction / composition を配置可能
   - ファセット解決順: プロジェクトローカル → orchestra（first-match-wins）

8. **パッケージフィルタ**: composition に `package` フィールドを指定し、対象パッケージが未インストールの場合はビルドをスキップ（既存生成物も自動削除）

## 影響

- **重複排除**: `code-quality` ファセットを1箇所で管理し、simplify スキルと coding-principles ルールの両方に反映
- **クロス CLI 配布**: 同じ composition から Claude Code / Codex 向けのスキルを自動生成。Gemini 対応は将来課題
- **SKILL.md / ルール .md が生成物**: facet build で生成されるファイルは直接編集せず、編集後は `facet extract` でソースに書き戻す運用
- **パッケージ連動**: 未インストールパッケージのスキル・ルールは配布先に生成されない
- **`orchex` CLI への反映**: facet サブコマンドは main マージ + PyPI 公開後に `orchex facet build` として利用可能。現時点では `python3 "$AI_ORCHESTRA_DIR/scripts/orchestra-manager.py" facet build` で直接実行
