# ADR-014: quality-gates に独立したテスト改ざん検出 Hook を追加

- **ステータス**: accepted
- **日付**: 2026-04-12
- **決定者**: yoshihiko
- **関連 Issue**: #4

## コンテキスト

`quality-gates` には lint 実行、テスト結果分析、変更量に応じたレビュー提案はあるが、AI がテストを通すために `it.skip()` や `eslint-disable` を追加するような「品質ゲートの迂回」を検出する仕組みがなかった。

Issue #4 では、JavaScript/TypeScript と Python の両方で skip / disable 系の追加を検知し、テストファイル削除も警告対象にすることが求められている。また、既存の `lint-on-save.py` と競合させず、`quality-gates` の責務として独立させる必要がある。

## 検討した選択肢

### 選択肢 A: `lint-on-save.py` に検出ロジックを内包する

- メリット: Hook 数が増えない
- デメリット: 整形・lint と品質ゲートの責務が混ざる。Bash 経由の削除検出を扱いにくい

### 選択肢 B: `quality-gates` に独立 Hook を追加する

- メリット: 検出責務を分離できる。Edit/Write と Bash の両方を一か所で扱える
- デメリット: manifest とテストの更新箇所が増える

### 選択肢 C: `audit` パッケージで差分監査として扱う

- メリット: 変更監査の責務には近い
- デメリット: 目的は運用監査ではなく、その場の品質警告であり、`quality-gates` から離れる

## 決定

**選択肢 B** を採用する。

`quality-gates` に `test-tampering-detector.py` を独立追加し、次の方針で実装する:

1. Edit/Write 後は `git diff` の追加行を優先し、新規ファイルだけは `tool_input` の内容をフォールバックとして扱う
2. Bash 後は `rm` / `git rm` を含むコマンドだけを対象にして、削除された追跡済みテストファイルを `git diff` から検出する
3. 検出対象は `it.skip()` / `test.skip()` / `describe.skip()` / `@pytest.mark.skip` / `@unittest.skip` / `eslint-disable` / `noqa` / `type: ignore` とする
4. 出力は `[Warning]` の追加コンテキストとし、変更理由とテスト担保方法の説明を促す

## 影響

- `quality-gates` の PostToolUse フックが 1 つ増える
- 既存の lint / review / test gate ロジックとは分離され、競合リスクを抑えられる
- `rm` や `git rm` による tracked test file の削除が品質ゲートで可視化される
- 追加差分ベースのため、既存コード中に以前からある skip / disable 記述は原則として再警告しない

## 残余リスク

- Bash 経由の削除検出は `rm` / `git rm` と一般的な `sh -c` / `bash -lc` ラッパーを対象にした best-effort 実装であり、`find ... -delete` や `python -c` のような任意の削除手段までは追わない
- テストファイル削除の警告は `git diff` 上の tracked file を前提にしているため、Git 管理外ファイルや Git 差分に現れない削除は検出対象外になる
