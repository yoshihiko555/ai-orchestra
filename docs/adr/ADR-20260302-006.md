# ADR-20260302-006: cocoindex v2: mcp-proxy による MCP 共有化とポート自動導出

- **ステータス**: accepted
- **日付**: 2026-03-02
- **決定者**: yoshihiko
- **関連**: cocoindex パッケージ

## コンテキスト

cocoindex パッケージは Claude Code / Codex CLI / Gemini CLI に cocoindex-code MCP サーバーを自動プロビジョニングする。v1 では各 CLI が独立して stdio でプロセスを起動するため、以下の問題があった:

- **SQLite ロック競合**: cocoindex-code 内部の SQLite に複数プロセスが同時アクセスしてロック競合が発生する
- **リソース浪費**: 同一 MCP サーバーのプロセスが CLI 数分だけ起動される
- **複数プロジェクト使用時のポート競合**: 固定ポートでは複数プロジェクトを同時に使用できない

## 検討した選択肢

### 選択肢 A: 各 CLI のプロセスを維持し、SQLite WAL モードで競合を緩和

- メリット: 既存アーキテクチャの変更が最小限
- デメリット: SQLite 側の対応に依存し、根本解決にならない

### 選択肢 B: mcp-proxy で HTTP 共有化（SessionStart 起動 + PID ファイル管理）

- メリット: 単一プロセスで SQLite 競合を根本解決
- メリット: 各 CLI が HTTP/SSE で接続するため、プロセス管理が簡潔
- デメリット: mcp-proxy への依存追加、PID ファイル管理の複雑さ

### 選択肢 C: 常駐デーモンとして MCP サーバーを運用

- メリット: セッションをまたいで再利用可能
- デメリット: デーモン管理の複雑さが大幅に増加

### ポート割り当て: 固定ポート vs hash-based 導出

- **固定ポート**: シンプルだが複数プロジェクト同時使用でポート競合が発生する
- **ポートスキャン**: 空きポートを探すが冪等性が崩れる（起動ごとにポートが変わり git diff が発生）
- **hash-based 導出**: project_dir の MD5 ハッシュでポートを決定論的に割り当て、冪等性を維持

## 決定

**選択肢 B + hash-based ポート導出** を採用。

具体的な決定:

1. **v2 (proxy) モードをオプトインで追加**: `proxy.enabled: false` をデフォルトにし、`.local.yaml` で有効化する（v1 を壊さない）
2. **mcp-proxy で HTTP 共有**: Claude Code / Gemini CLI は `/sse`（SSE）、Codex CLI は `/mcp`（Streamable HTTP）で接続
3. **SessionStart で起動、SessionEnd で停止**: PID ファイルベースのライフサイクル管理（`proxy_manager.py` として分離）
4. **hash-based ポート導出**: `MD5(project_dir)[:4] % port_range` で `base_port` 〜 `base_port + port_range - 1` の範囲内で決定論的に割り当て（デフォルト: 8792〜8891）
5. **`port_range: 0` で固定モード**: 明示的にポートを指定したい場合のエスケープハッチ
6. **`force_stdio` フラグ**: proxy 有効時でも特定 CLI を stdio に強制可能
7. **`--port` + `--host` を使用**: mcp-proxy v0.11.0 で `--sse-port` は deprecated

## 影響

- 複数 CLI の同時使用時に SQLite ロック競合が解消される
- 複数プロジェクトの同時使用がポート競合なしで可能になる
- mcp-proxy（`uv tool install mcp-proxy`）が新たな依存として追加される
- v1 ユーザーへの影響なし（デフォルト `proxy.enabled: false`）
- `cocoindex-usage.md` に v2 の設定方法・トラブルシューティングを追記済み
