# ADR-20260419-016: quality gate の判定は quality-gates が担い、audit は記録と集計に限定する

- **ステータス**: accepted
- **日付**: 2026-04-19
- **決定者**: AI Orchestra maintainers

## コンテキスト

`audit` パッケージには `quality_gate` event の記録と `block_on_failed_test` の制御が同居していた。
一方で、実際の品質ゲート判定やテスト後の提案は `quality-gates` パッケージが担っている。

この構成には次の問題があった。

- `audit` が観測だけでなく制御責務まで持っていた
- `audit-route.py` は通常の `pytest -q` のような Bash テスト実行を route 非該当として早期 return し、`quality_gate` event を記録できていなかった
- 責務境界が曖昧で、`quality_gate` の不具合箇所が分かりづらかった

## 検討した選択肢

### 選択肢 A: `audit` が route 記録と quality gate 制御の両方を持ち続ける

- メリット:
  - 既存ファイルへの変更が少ない
  - `quality_gate` event を `audit` 側だけで完結できる
- デメリット:
  - 観測と制御の責務が混ざる
  - テスト後解析を `quality-gates` が持っている現状と整合しない
  - 今回のように通常 Bash テスト実行を取りこぼしやすい

### 選択肢 B: `quality-gates` が品質ゲート判定を行い、`audit` は event の集計と可視化に専念する

- メリット:
  - `quality-gates` が「判定 / 提案 / block」の責務を一元的に持てる
  - `audit` は event 記録・KPI・ダッシュボードに集中できる
  - `post-test-analysis.py` はテスト結果を直接見ているため、`quality_gate` event の記録元として自然
- デメリット:
  - `quality-gates` から `audit` の event logger を利用する実装が必要
  - 共有設定の所在を明確にドキュメント化する必要がある

## 決定

選択肢 B を採用する。

- `audit-route.py` は `route_decision` の記録に限定する
- `quality-gates/post-test-analysis.py` が `quality_gate` event を記録する
- `block_on_failed_test` の判定も `quality-gates` 側に移す
- `audit` は `quality_gate` event を集計・可視化するが、判定ロジックは持たない

## 影響

- `quality_gate` event の記録元は `audit-route.py` から `post-test-analysis.py` へ移る
- `quality-gates` の `audit` 依存は、共有設定だけでなく event 記録の観点でも必須になる
- `audit-flags.json` の `quality_gate` セクションは、`audit` と `quality-gates` の共有設定として扱う
- `essential` preset では `quality-gates` の依存を満たすため `audit` を含める
