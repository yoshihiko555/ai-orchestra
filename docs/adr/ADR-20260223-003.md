# ADR-20260223-003: task-memory サマリー表示仕様の明確化と marker 設定の堅牢化

- **ステータス**: accepted
- **日付**: 2026-02-23
- **決定者**: yoshihiko

## コンテキスト

`packages/task-memory/hooks/load-task-state.py` の SessionStart サマリーは運用に使われている一方、仕様と実装の不一致があった。

- `max_display_tasks: 0`（無制限）が実装で 20 にフォールバックされ、設定コメントと不一致
- `max_display_tasks` が「合計上限」ではなく、実質的にセクションごと表示になっていた
- `markers` 設定が未使用で、マーカー拡張が設定からできなかった
- marker 値の重複設定ミスを検知できず、状態分類の誤りが静かに発生し得た
- タスク本文が空の行（例: `- \`cc:TODO\``）が有効タスクとして集計され得た
- `AI_ORCHESTRA_DIR` 未設定時に `hook_common` を読めず、設定上書きが効かないケースがあった

## 検討した選択肢

### 選択肢 A: 最小バグ修正のみ（`max_display_tasks=0` 対応のみ）

- メリット: 変更範囲が小さい
- デメリット: 仕様不整合（合計上限/marker設定/空タスク/設定読込）が残る

### 選択肢 B: サマリー仕様を明文化し、設定駆動化とバリデーションを同時に導入

- メリット: 表示ルールと設定意図が一致し、運用時の予測可能性が高まる
- メリット: marker 拡張が可能になり、設定ミス時の安全なフォールバックが入る
- メリット: 空タスク混入や環境依存設定ロードの不安定さを解消できる
- デメリット: 既存の表示順（WIP→blocked→TODO）に依存した利用者への周知が必要

### 選択肢 C: marker 設定不正時に fail-fast（サマリー出力を停止）

- メリット: 設定ミスを強く顕在化できる
- デメリット: SessionStart でサマリー自体が消え、運用影響が大きい

## 決定

**選択肢 B** を採用。

具体的な決定:
1. `max_display_tasks` を **サマリー全体の合計上限** と定義し、`0` は無制限とする
2. 表示優先度を **WIP → TODO → blocked** に変更（TODO を blocked より優先表示）
3. `markers` 設定から parser 用 regex を組み立てる設定駆動方式に変更
4. marker 値の重複を検知し、重複時は警告を出してデフォルト marker にフォールバック
5. マーカー後の本文が空のタスク行は集計対象から除外
6. `AI_ORCHESTRA_DIR` 未設定時もリポジトリ内 `packages/core/hooks` を探索して設定ロードを継続
7. 変更を回帰固定するため、`packages/task-memory/tests/test_load_task_state.py` に単体テストを追加・拡張

## 影響

- `packages/task-memory` の表示結果は「合計上限」と「WIP→TODO→blocked」前提になる
- 上限により blocked が表示されない場合でも、省略された事実がサマリーに明示される
- marker 設定の誤り（重複）は silent failure にならず、stderr 警告 + 安全フォールバックになる
- 空タスク行が除外され、サマリー品質が向上する
- `AI_ORCHESTRA_DIR` 依存が緩和され、プロジェクトローカル設定の適用安定性が上がる
- 新規テストにより `load_config` / `parse_tasks` / `format_summary` / `main` の仕様を継続的に検証できる
