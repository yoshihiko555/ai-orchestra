# ADR-20260223-003: Codex CLI 実装委譲のデッドロック解消と Implementation Method 強制

- **ステータス**: accepted
- **日付**: 2026-02-23
- **決定者**: yoshihiko
- **関連**: ADR-20260219-002（startproject 実装フェーズの明示化）

## コンテキスト

ADR-002 で startproject の Implementation Phase 明示化と Codex 例外の厳格化を行ったが、実装系エージェント（`frontend-dev`, `backend-python-dev` 等）が `cli-tools.yaml` で `tool: codex` に設定されているにもかかわらず、Codex CLI を一切使用せず Claude が直接 Edit/Write で実装してしまう問題が残っていた。

根本原因は 3 つ:

1. **Hook + Compliance Rule のデッドロック**: PreToolUse hook が `[Codex Suggestion]` を出力 → compliance rule が Codex 相談を強制 → Codex CLI はサンドボックスで動作不可 → 実装不能のループ
2. **エージェント指示が弱い**: agent .md の CLI Usage セクションは条件付き記述で、Edit/Write 直接使用を明示的に禁止していなかった
3. **startproject Agent Routing テーブルの矛盾**: `tool: codex` のエージェントを `general-purpose` 経由で呼び出すよう指定していたが、Phase 6 例では named agent を直接呼び出しており不整合

## 検討した選択肢

### 選択肢 A: Compliance Rule に例外追加 + エージェント指示強化 + スキル修正

- メリット: 3 つの根本原因すべてに対処できる
- メリット: hook ロジック変更不要（compliance rule 側で対処）
- メリット: named agent を直接呼び出すことでドメイン知識が保持される
- デメリット: 8 エージェントファイル + 2 スキルファイルの一括変更が必要

### 選択肢 B: hook（check-codex-before-write.py）にサブエージェント検出を追加

- メリット: 技術的に正確な判定が可能
- デメリット: PreToolUse の入力データにサブエージェント情報が含まれないため実装不可能

### 選択肢 C: `tool: codex` のエージェントを `tool: claude-direct` に変更

- メリット: デッドロックが即座に解消
- デメリット: Codex CLI の活用を完全に放棄することになり、本末転倒

## 決定

**選択肢 A** を採用。4 つの変更を実施:

### 1. Compliance Rule に例外追加（デッドロック解消）

`codex-suggestion-compliance.md` の例外セクションに `tool: codex` エージェント内での Edit/Write を追加。これにより、implementation agent が Codex CLI 経由で実装する際に `[Codex Suggestion]` hook がデッドロックを引き起こさなくなる。フォールバック時（`codex.enabled: false`）のみ Edit/Write 直接使用を許可。

### 2. 8 エージェントの Implementation Method 強制

`Configuration` + `CLI Usage` セクションを `Configuration` + `Implementation Method（必須）` に置換:
- `tool = "codex"` 時の Edit/Write 禁止を明記
- `[Codex Suggestion]` hook の無視許可を明記
- `dangerouslyDisableSandbox: true` を Codex コマンド直前に明記
- `<codex.model>` 等の具体的なプレースホルダーを使用
- フォールバックルールを統合

対象: `frontend-dev`, `backend-python-dev`, `backend-go-dev`, `ai-dev`, `tester`, `debugger`, `rag-engineer`, `spec-writer`

### 3. startproject Agent Routing テーブル修正

`tool: codex` の呼び出し方法を `general-purpose` 経由から named agent 直接呼び出しに変更。Phase 6 プロンプト例に Codex CLI 指示を明記。

### 4. issue-fix Phase 2-2 にエージェント委譲ガイド追加

3 箇所以上の変更は implementation agent に委譲、1-2 箇所の軽微な修正はオーケストレーター直接実行を許可。

## 影響

- `tool: codex` エージェントで Codex CLI が確実に使用されるようになる
- `[Codex Suggestion]` hook と compliance rule のデッドロックが解消される
- named agent の直接呼び出しにより、ドメイン固有の知識（コーディング規約、テックスタック）が保持される
- `codex.enabled: false` 時は claude-direct にフォールバックし、柔軟性を維持
- 検証で `backend-python-dev` が正しく cli-tools.yaml を読み → Codex CLI を実行する動作を確認済み
