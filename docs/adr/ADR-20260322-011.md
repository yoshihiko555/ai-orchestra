# ADR-20260322-011: manifest-SSOT アーキテクチャ（facets 正本化・パッケージ skills 廃止）

- **ステータス**: accepted
- **日付**: 2026-03-22
- **決定者**: yoshihiko
- **関連**: Issue #20, facet composition システム, ADR-20260315-010

## コンテキスト

facet composition システム導入後、スキルの正本が二重管理になっていた。

- `facets/compositions/**/*.yaml` + `facets/instructions/*.md` が実質的な正本
- `packages/*/skills/*/SKILL.md` が中間生成物として存在
- `manifest.json` の `skills` フィールドと `composition.yaml` の `package` フィールドが所有者情報を二重管理
- 直接 `packages/.../SKILL.md` を編集すると次回 facet build で上書きされる

管理形態も分断されていた:

- facets が skills/rules の内容を管理
- manifest.json が hooks/config/agents を管理
- パッケージの全体像を把握するには両方を見る必要があった

## 検討した選択肢

### 選択肢 A: manifest.json を SSOT に（採用）

manifest.json でパッケージの全リソース（skills, rules, hooks, config, agents）を一元管理する。
composition.yaml の `package` フィールドを廃止し、所有者は manifest から逆引きする。

- メリット: manifest.json を見ればパッケージの全容がわかる
- メリット: composition はビルドレシピに専念でき責務が明確
- メリット: install/uninstall のロジックがシンプル
- デメリット: ビルド時に全パッケージの manifest をスキャンする必要がある

### 選択肢 B: facets を SSOT に

composition.yaml に hooks/config も定義し、manifest.json を自動生成または廃止する。

- メリット: facets に全て集約され一箇所で管理
- デメリット: hooks/config の定義が YAML 内に分散し見通しが悪い
- デメリット: manifest.json の互換性を壊す大規模な変更が必要

## 決定

**選択肢 A（manifest-SSOT）を採用**。

具体的な変更:

1. `manifest.json` の `skills`/`rules` を composition 名形式に変更（`"skills/preflight"` → `"preflight"`）
2. `composition.yaml` から `package` フィールドを削除
3. `packages/*/skills/` ディレクトリを全削除（SKILL.md は facet build でのみ生成）
4. `FacetBuilder` が `manifest_compositions`（全パッケージ）+ `installed_packages` でビルド対象を判定
5. `sync_packages()` から skills カテゴリを除外（facet build に完全委譲）
6. グローバル composition（どの manifest にも含まれない）は常にビルド

## 影響

- `packages/*/skills/` は存在しなくなる。スキルの実体は facet build の出力のみ
- 新しい composition を作成したら、対応するパッケージの `manifest.json` に composition 名を追加する必要がある
- uninstall 時のファイル削除が composition 名形式に対応（skills はディレクトリ、rules は `.md` 付きで削除）
- `collect_manifest_compositions()` が全パッケージの manifest をスキャンしてマッピングを構築する
- 詳細設計: `.claude/docs/design/manifest-ssot-architecture.md`
