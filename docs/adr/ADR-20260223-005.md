# ADR-20260223-005: Codex/Gemini 運用記述の config-driven 統一

- **ステータス**: accepted
- **日付**: 2026-02-23
- **決定者**: yoshihiko
- **関連**: ADR-20260223-003, Issue #13

## コンテキスト

Codex/Gemini の責務がドキュメント上で固定的に記述されており、実際の `cli-tools.yaml` ルーティングと齟齬が発生していた。

- `codex-system` / `gemini-system` / delegation rule / usage guide の説明が「設計は Codex」「調査は Gemini」など固定役割前提
- config 参照パスに旧表記（`.claude/config/cli-tools.yaml`）が残存
- 現在は `agents.<name>.tool` と `tool: auto` による切り替え運用を採用しており、将来のモデル評価変化に合わせて役割変更する可能性がある

## 検討した選択肢

### 選択肢 A: 既存の固定役割記述を維持

- メリット: 文書上の説明が短い
- デメリット: `cli-tools.yaml` と乖離しやすく、運用誤解を生む

### 選択肢 B: `cli-tools.yaml` を SSOT とする config-driven 記述に統一

- メリット: 実運用（`agents.<name>.tool`）と文書が一致する
- メリット: Codex/Gemini の役割変更を設定変更のみで追従できる
- デメリット: 文書の説明がやや長くなる

### 選択肢 C: ツール役割の説明をほぼ削除し、運用者判断に委ねる

- メリット: 文書保守量が減る
- デメリット: 初見利用者の判断コストが上がる

## 決定

**選択肢 B** を採用。

具体的な決定:
1. Codex/Gemini 関連スキル・ルール・ガイドの記述を `cli-tools.yaml` 参照前提へ統一する
2. `agents.<target>.tool` を一次判定とし、`tool: auto` の場合のみヒューリスティクスを適用する
3. 旧 config パス表記を `.claude/config/agent-routing/cli-tools.yaml` に統一する
4. README/CLAUDE の概要説明も固定責務ではなく config-driven 表現に更新する

## 影響

- Codex/Gemini の役割変更時に、設定変更だけで文書の意図と整合が取りやすくなる
- エージェント説明と実際のルーティングの誤解が減る
- なお、`design-tracker` と migration-guide の残課題は Issue #13 で別管理する
