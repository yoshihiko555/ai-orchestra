# ADR-20260308-008: サブエージェント model の config-driven 自動パッチ

- **ステータス**: accepted
- **日付**: 2026-03-08
- **決定者**: yoshihiko
- **関連**: agent-routing パッケージ, ADR-20260223-005

## コンテキスト

全 25 エージェントの `.md` ファイルに `model: sonnet` がハードコードされており、モデルを変更するには全ファイルを手動編集する必要があった。プロジェクトごとに異なるモデルを使いたい場合や、特定エージェントだけ高性能モデルを割り当てたい場合に運用負荷が高かった。

## 検討した選択肢

### 選択肢 A: エージェント実行時に動的解決

- メリット: ファイル書き換えが不要
- デメリット: Claude Code の Agent tool は frontmatter の `model` フィールドを静的に読み取るため、動的解決の手段がない

### 選択肢 B: sync-orchestra.py で SessionStart 時に frontmatter を自動パッチ

- メリット: 既存の同期基盤を活用でき、設定変更が即座に全エージェントに反映される
- メリット: cli-tools.yaml / cli-tools.local.yaml のレイヤード構成をそのまま利用可能
- デメリット: パッチにより dst の mtime がソースより新しくなり、ソース側の model 以外の変更が次セッションでコピーされないリスクがある（ただしパッチは毎セッション全エージェントに試みるため model 値は常に最新）

### 選択肢 C: 専用のモデル設定ファイルを新設

- メリット: 既存ファイルへの副作用がない
- デメリット: 新しい設定レイヤーが増え複雑化する。Claude Code が参照する frontmatter との二重管理になる

## 決定

**選択肢 B: sync-orchestra.py で SessionStart 時に frontmatter を自動パッチ**する。

理由:

1. **既存基盤の活用**: sync-orchestra.py の同期フローに自然に組み込める
2. **config-driven の一貫性**: ADR-005 で確立した cli-tools.yaml 一元管理の方針に沿う
3. **2 層の優先順位**: `agents.<name>.model`（per-agent）> `subagent.default_model`（グローバル）で柔軟に制御可能
4. **pyyaml 未導入時の安全性**: yaml import 失敗時はパッチをスキップし、既存の同期処理に影響しない

## 影響

- `cli-tools.yaml` の `subagent.default_model` で全エージェントのデフォルトモデルを一括変更可能
- `agents.<name>.model` で特定エージェントのみ別モデルを指定可能（`null` はデフォルトに従う）
- `cli-tools.local.yaml` でプロジェクト固有のモデル設定を上書き可能
- パッチは値が同一の場合スキップされるため、不要な書き換えは発生しない
