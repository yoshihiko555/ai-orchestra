# ADR-20260313-009: CLI 間コンテキスト共有のファイルベース設計

- **ステータス**: accepted
- **日付**: 2026-03-13
- **決定者**: yoshihiko
- **関連**: core パッケージ, ADR-20260223-005, ADR-20260302-006

## コンテキスト

Claude Code / Codex CLI / Gemini CLI の 3 CLI 間でコンテキストが共有されておらず、Codex や Gemini を呼び出す際に必要な文脈をすべて prompt に手動で埋め込む必要があった。また、並列サブエージェント間で互いの発見・結果をリアルタイムに参照する仕組みがなかった。

ai-valification リポジトリでの先行検証により、以下が確認済みだった:
- Map-Reduce パターンによる JSON ファイルベースの並列サブエージェント間共有（検証済み）
- サブエージェントへの命令伝達パス（セッション開始時キャッシュ、途中変更は非伝播）
- Bash + subagent 方式が MCP 方式よりコンテキスト効率が高い（Issue #6915 待ち）

## 検討した選択肢

### 選択肢 A: ファイルベース（`.claude/context/` に JSON ファイル）

- メリット: 3 CLI すべてが native にアクセス可能（Read/Write ツール、Bash の cat/jq）
- メリット: デバッグが容易（ファイルを直接確認できる）
- メリット: ai-valification で検証済みのパターンを拡張するだけで低リスク
- メリット: プロジェクトスコープで管理でき、クリーンアップが容易
- デメリット: リアルタイム通知がない（ポーリングまたは inotify が必要）

### 選択肢 B: 専用 MCP サーバー（cocoindex パターン）

- メリット: 構造化された API でアクセス可能（JSON-RPC ベース）
- メリット: リアルタイム通知が可能
- デメリット: MCP ツール定義が全エージェントのコンテキストに常駐しトークンを圧迫（Issue #6915 未解決）
- デメリット: サーバープロセスの管理が必要（cocoindex proxy と同様の運用負荷）
- デメリット: Codex/Gemini からは結局 Bash 経由の curl になり、prompt 埋め込みと変わらない

### 選択肢 C: JSON-RPC サーバー（MCP 非依存）

- メリット: MCP のコンテキストコストなしに構造化 API が使える
- デメリット: サーバープロセスの管理が必要
- デメリット: CLI からのアクセスは Bash 経由の curl（ファイルアクセスより複雑）
- デメリット: MCP が既に JSON-RPC ベースの通信層を担っており、機能が重複する

## 決定

**選択肢 A: ファイルベース（`.claude/context/`）** を採用する。

理由:

1. **3 CLI の共通アクセス**: ファイルシステムは全 CLI が native にアクセスできる唯一の共通インターフェース
2. **検証済みパターンの活用**: ai-valification で検証済みの Map-Reduce パターンを本番化するだけで低リスク
3. **MCP コンテキストコスト回避**: Issue #6915（`subagentOnly` フラグ）が解決するまで MCP ツール追加はトークン圧迫が大きい
4. **プロジェクトスコープ管理**: `/tmp/` ではなく `.claude/context/` に配置し、プロジェクトごとに独立管理
5. **セッション間記憶の委任**: claude-mem が既にセッション間の記憶永続化を担当しており、独自の履歴管理は不要

### ストレージ構造

```
.claude/context/
  session/                    # セッションスコープ（終了時クリーンアップ）
    meta.json                 # セッション ID、開始時刻
    entries/                  # サブエージェント結果（Map-Reduce）
      {agent_id}.json
  shared/                     # CLI 間共有
    working-context.json      # 作業中ファイル・設計判断・フェーズ
```

### 実装方針

- `packages/core/hooks/context_store.py` に共通ユーティリティを集約
- 既存 hook（SessionStart, PreToolUse, PostToolUse）を拡張し、新規パッケージは作成しない
- セッション間記憶は claude-mem に委任し、`history/` ディレクトリは設けない

## 影響

- 並列サブエージェントが `session/entries/` 経由で互いの結果を参照可能になる
- Codex/Gemini 委譲時に `working-context.json` から作業コンテキストが prompt に自動注入される
- SessionEnd で `session/` がクリーンアップされ、永続記憶は claude-mem が担当する
- 将来 MCP の `subagentOnly` フラグが実装された場合、MCP サーバー方式への移行パスが残る
