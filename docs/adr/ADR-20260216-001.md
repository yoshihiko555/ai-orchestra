# ADR-20260216-001: docs ディレクトリのカテゴリ分類と ADR 導入

- **ステータス**: accepted
- **日付**: 2026-02-16
- **決定者**: yoshihiko

## コンテキスト

`docs/` 配下が未整理の状態だった。

- 空テンプレート（`DESIGN.md`）や空ディレクトリ（`libraries/`, `research/`）が混在
- 仕様書・ガイド・レポートが同一階層にフラットに配置され、種別が不明瞭
- ADR（意思決定記録）の仕組みがなく、設計判断の経緯が追跡不能

## 検討した選択肢

### 選択肢 A: カテゴリ別サブディレクトリ + ADR 導入

- メリット: 種別ごとにファイルが整理され、新規ドキュメントの配置先が明確になる
- メリット: ADR により設計判断の経緯を記録・追跡できる
- デメリット: 既存のパス参照を更新する必要がある

### 選択肢 B: 現状維持 + ファイル名規約のみ

- メリット: パス変更なし、影響範囲ゼロ
- デメリット: ファイル数が増えるとフラット構造では管理が難しくなる
- デメリット: ADR の仕組みがないまま

## 決定

**選択肢 A** を採用。カテゴリ別に `specs/`, `guides/`, `reports/`, `adr/` を新設し、既存ファイルを移動。配布先プロジェクト用の空テンプレート・空ディレクトリは削除。

構造:
- `docs/adr/` — 意思決定記録
- `docs/specs/` — 仕様書
- `docs/guides/` — 運用・手順ドキュメント
- `docs/reports/` — レポート
- `docs/comparisons/` — KPI レポート（変更なし）
- `docs/tech/` — 技術調査（.gitignore 済み、変更なし）

## 影響

- `README.md` のパス参照を `docs/logging.md` → `docs/specs/logging.md` に更新
- 今後のドキュメント追加時は適切なサブディレクトリに配置する
- 設計判断は `docs/adr/` に ADR として記録する運用を開始
